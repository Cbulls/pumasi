# 품앗이폼 — 결제(크레딧 현금 구매) 설계서 v1.0

> ⚠️ 이 모듈은 PG사 계약 전이라 **실제 구매 기능은 미구현**일 수 있다. 이 설계서는
> 계약 없이 지금 작성 가능한 부분과, 계약 후에만 채울 수 있는 부분을 명확히 구분한다.
> 핵심 전략: 결제 도메인을 PG에 의존하지 않게 추상화 → 계약 후 어댑터만 끼운다.

---

## 0. 왜 이렇게 설계하나 (PG 비의존)

사업자등록 → PG 가입·심사 → 계약 → API 키 발급을 거쳐야 실제 결제가 가능하다. 그 전에도
**결제 도메인 로직은 100% 완성하고 검증할 수 있어야 한다.** 그래야 계약 후 작업이 어댑터
한 곳에 격리되고, 도메인은 손대지 않는다.

방법: 헥사고날 아키텍처(포트 & 어댑터).
- 도메인(PaymentService)은 `PaymentGateway` 인터페이스(포트)에만 의존.
- 실제 PG(토스/포트원)는 이 포트의 구현체(어댑터)로만 존재.
- 계약 전: `FakePaymentGateway`로 전체 흐름 검증(11 시나리오 통과 ✅).
- 계약 후: `TossPaymentsGateway`만 채우면 됨. 도메인 무수정.

## 2. 결제 3대 위험과 방어 (도메인에 내장)

결제는 돈이 오가므로 고유 위험이 있다. 방어를 어댑터가 아니라 **도메인에 내장**해서,
PG를 무엇으로 바꾸든 방어가 작동하게 한다.

| 위험 | 시나리오 | 방어 |
|---|---|---|
| ★위변조★ | 클라가 "5000원→10원" 조작 | 금액을 서버가 패키지 ID로 결정. 승인 후 PG 실제 금액 재대조. 불일치면 적립 거부 + PG 취소 |
| ★이중지급★ | 네트워크 재시도로 두 번 적립 | orderId 멱등 + 상태기계(CREATED→APPROVED→CREDITED 1회). 크레딧 원장 (reason,ref_id) 유니크 |
| ★상태불일치★ | PG 승인됐는데 적립 실패 | 승인+적립을 한 트랜잭션. 적립 실패 시 PG 취소(보상 트랜잭션) 후 롤백 |

> 검증: FakePaymentGateway로 5개 시나리오(정상/위변조/이중지급/상태불일치/카드거부)
> 모두 방어 확인. 정상 적립 550, 위변조 시 미적립+PG취소, 재시도 시 1회만 적립 등.

## 3. 결제 흐름 (2단계 모델)

대부분의 국내 PG가 공유하는 모델. "결제됨 ≠ 승인됨"이 핵심.

```
1단계 (클라이언트):
  GET  /api/payments/packages          패키지 선택
  POST /api/payments/orders            주문 생성 → orderId, amount(서버 결정)
  PG 결제창 SDK 호출 → 사용자 인증·결제 → paymentKey 수신

2단계 (서버):
  POST /api/payments/confirm           orderId + paymentKey
   → PaymentService.confirmPayment:
       PG 승인(approve) → 금액 대조 → markApproved
       → 크레딧 적립(멱등) → markCredited
```

서버가 2단계 승인(confirm)을 직접 해야 금액 위변조를 막는다. 클라이언트가 "결제 완료"라고
주장하는 것만으로는 절대 크레딧을 주지 않는다 — 서버가 PG에 직접 확인한다.

## 4. 컴포넌트 구조

```
com.pumasiform.payment
├── PaymentGateway          포트 (PG 중립 인터페이스) — 도메인이 의존하는 유일한 PG 접점
│   · approve / cancel / providerName
├── PaymentTypes            값 객체(ApproveCommand, PaymentApproval...) + PaymentException
├── PaymentOrder            주문 엔티티(상태기계 + 멱등)
├── PaymentService          도메인 규칙(주문 생성, 승인, 적립, 3대 위험 방어)
├── CreditPackageCatalog    패키지 정의(금액·크레딧을 서버가 소유 — 위변조 차단 근거)
├── CreditChargePort        크레딧 적립 포트(크레딧 모듈로 의존 역전)
├── PaymentController       REST API
├── FakePaymentGateway      [계약 전] 어댑터 — @Profile("!prod"), 전체 흐름 검증
└── TossPaymentsGateway     [계약 후] 어댑터 — @Profile("prod"), ◆ 표시가 채울 자리
```

### Profile로 어댑터 전환
- 개발/테스트: `FakePaymentGateway`(@Profile("!prod"))가 빈으로 등록.
- 운영: `TossPaymentsGateway`(@Profile("prod"))가 등록.
- PaymentService는 어느 쪽이 주입되는지 모른다(포트만 안다).

## 5. ◆ 계약 후 채울 자리 (TossPaymentsGateway)

PG사 계약·키 발급 후에만 가능한 작업. 모두 어댑터 한 파일 + 웹훅 핸들러에 격리된다.

1. **시크릿 키 / 엔드포인트**: 환경변수·시크릿 매니저로 주입(코드·git 금지).
   토스 승인 URL: `https://api.tosspayments.com/v1/payments/confirm`
2. **approve() 구현**: PG 승인 API 호출 → 응답을 `PaymentApproval`(중립 모델)로 변환.
   PG 에러코드 → `PaymentException.Code` 매핑(DECLINED/GATEWAY_ERROR/AMOUNT_MISMATCH...).
3. **cancel() 구현**: 취소·환불 API.
4. **웹훅 핸들러**: PG 비동기 통보(가상계좌 입금 등). 서명 검증(HMAC) + 멱등(중복 통보).
   `payment_webhook_log` 테이블에 수신 기록.
5. **클라이언트 키**: 프론트 결제위젯 SDK 초기화용 공개 키(시크릿과 별개).
6. **정산/세금계산서**: PG 대시보드 또는 정산 API(회계).
7. **PCI-DSS**: 카드 정보는 절대 우리 서버를 거치지 않게. PG 결제창이 직접 처리.
