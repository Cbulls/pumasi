# 품앗이폼 → 전자정부표준프레임워크 전환 설계서 (학습용) v1.0

> 개인 학습 목적. 현재 Spring Boot(도메인 중심·헥사고날·JPA) 구조를
> 전자정부표준프레임워크(eGovFrame) 규약으로 전환하는 설계.
> 단순 변환 매뉴얼이 아니라, **두 구조가 무엇이 왜 다른지**를 대조하며 표준프레임워크의
> 사상을 이해하는 것을 목표로 한다.

---

## 0. 먼저 이해할 것: 표준프레임워크는 "Spring + 규약"이다

전자정부표준프레임워크는 새 프레임워크가 아니라, **Spring에 정부 표준 규약과 공통 컴포넌트를
얹은 것**이다. 그래서 전환은 "기술을 바꾸는" 게 아니라 "규약을 따르는" 작업에 가깝다.

표준프레임워크가 강제하는 핵심 규약(호환성 가이드 기준):
1. **실행환경 라이브러리**: `org.egovframework.rte.*` (구버전 `egovframework.rte`)를 쓰고,
   인위적 변경 금지(확장은 상속으로).
2. **서비스 아키텍처 규칙**: 서비스 클래스가 표준프레임워크가 정한 구조를 따라야 한다
   (인터페이스 + ServiceImpl, `EgovAbstractServiceImpl` 상속).
3. **계층형 아키텍처**: Controller → Service(인터페이스) → ServiceImpl → DAO → MyBatis.
4. **DI 스타일**: `@Resource(name=...)` 이름 기반 주입(전통적 관례).

> 학습 포인트: 우리가 쓴 "현대 Spring Boot 관례"와 표준프레임워크의 "전통 Spring 관례"는
> 둘 다 Spring이지만 시대가 다르다. 표준프레임워크를 공부하면 2010년대 한국 SI의
> 표준 사고방식을 익히는 셈이다.

## 1. 구조 대조 (현재 vs 표준프레임워크)

| 항목 | 현재(품앗이폼) | 표준프레임워크 | 학습 메모 |
|---|---|---|---|
| 아키텍처 | 도메인 중심 + 헥사고날(포트&어댑터) | 계층형(Controller/Service/DAO) | 표준은 계층형이 기본 |
| 패키지 | `com.pumasiform.{module}` | `egovframework.{업무}.{기능}` | 패키지 규약부터 다름 |
| 서비스 | `@Service` 클래스 1개 | `Service`(인터페이스) + `ServiceImpl` | 인터페이스 강제 |
| 서비스 상속 | 없음(POJO) | `EgovAbstractServiceImpl` 상속 | 표준의 핵심 요건 |
| 영속성 | JPA(`@Entity` + `JpaRepository`) | MyBatis(DAO + Mapper XML) | 가장 큰 전환 작업 |
| DAO | Spring Data 인터페이스 | `EgovAbstractDAO`/`EgovAbstractMapper` 상속 | 인증 요건 |
| DI | 생성자 주입 | `@Resource(name=...)` 필드 주입 | 관례 차이 |
| ID 채번 | UUID/DB | `EgovIdGnrService`(채번 서비스) | 표준 공통 서비스 |
| 예외 | 커스텀 RuntimeException | `EgovBizException` 등 | 표준 예외 체계 |
| 검증 | 커스텀 + Bean Validation | Jakarta Validation + 표준 가이드 | 유사 |
| 페이지네이션 | cursor/커스텀 | `PaginationInfo`(표준 제공) | 공통 컴포넌트 |

## 2. 변환 원칙: 비즈니스 로직은 살린다

가장 중요한 전략. 우리는 비즈니스 로직을 **순수 Java로 프레임워크와 분리**해뒀다
(FormValidator, QualityJudge, CreditConcurrency, BaselineCalculator 등 250 테스트).

→ 이 순수 로직은 **그대로 두고**, 그걸 호출하는 서비스/DAO 계층만 표준프레임워크 규약으로
다시 감싼다. 프레임워크 분리 설계가 여기서 값을 한다.

```
[현재]  Controller → @Service(포트 주입) → 순수 로직 + JpaRepository
[전환]  Controller → Service(if) → ServiceImpl(EgovAbstractServiceImpl) → 순수 로직 + DAO(MyBatis)
                                         ↑ 여기만 다시 쓴다          ↑ 살린다  ↑ JPA→MyBatis
```

## 3. 변환 예제: 폼 빌더 (한 모듈 끝까지)

폼 빌더 하나를 끝까지 변환해 패턴을 익힌다. 나머지 모듈은 같은 패턴의 반복.

### 3.1 패키지 재구성
```
[현재]  com.pumasiform.form.{Form, FormService, FormController, FormRepository}
[전환]  egovframework.pmsi.form.web.EgovFormController
        egovframework.pmsi.form.service.FormService          (인터페이스)
        egovframework.pmsi.form.service.impl.FormServiceImpl  (EgovAbstractServiceImpl 상속)
        egovframework.pmsi.form.service.impl.FormDAO          (EgovAbstractMapper 상속)
        egovframework.pmsi.form.service.FormVO                (VO — DTO 대체)
        resources/egovframework/mapper/pmsi/form/Form_SQL.xml (MyBatis 매퍼)
```
> `pmsi` = 품앗이폼 업무 약칭(표준은 업무별 패키지 분리). web/service/service.impl 3계층.

### 3.2 Service 인터페이스 (신규 — 표준은 인터페이스 강제)
```java
package egovframework.pmsi.form.service;

public interface FormService {
    String createForm(FormVO vo) throws Exception;   // 표준은 throws Exception 관례
    FormVO selectForm(String formId) throws Exception;
    void addQuestion(QuestionVO vo) throws Exception;
}
```

### 3.3 ServiceImpl (EgovAbstractServiceImpl 상속 + 순수 로직 재사용)
```java
package egovframework.pmsi.form.service.impl;

import egovframework.rte.fdl.cmmn.EgovAbstractServiceImpl;   // ★ 표준 핵심
import egovframework.rte.fdl.idgnr.EgovIdGnrService;
import org.springframework.stereotype.Service;
import javax.annotation.Resource;

@Service("formService")
public class FormServiceImpl extends EgovAbstractServiceImpl implements FormService {

    @Resource(name = "formDAO")                 // ★ 이름 기반 주입
    private FormDAO formDAO;

    @Resource(name = "egovFormIdGnrService")    // ★ 표준 채번 서비스
    private EgovIdGnrService idgenService;

    // 우리가 만든 순수 검증 로직을 그대로 재사용(프레임워크 무관)
    private final FormValidator validator = new FormValidator();

    @Override
    public void addQuestion(QuestionVO vo) throws Exception {
        // 검증은 검증된 순수 로직으로 — 이 부분은 안 바뀐다
        List<String> errors = validator.validate(toQuestionMap(vo));
        if (!errors.isEmpty()) {
            // 표준 예외 체계 사용
            throw processException("form.validation.fail");
        }
        vo.setQuestionId(idgenService.getNextStringId());  // 표준 채번
        formDAO.insertQuestion(vo);                        // DAO 위임
    }
}
```
> 학습 포인트: `EgovAbstractServiceImpl`을 상속하면 `processException()`(표준 예외),
> `leaveaTrace()`(로깅) 등을 쓸 수 있다. 이 상속이 "표준프레임워크 준수"의 핵심 증표다.

### 3.4 DAO (EgovAbstractMapper 상속 + MyBatis)
```java
package egovframework.pmsi.form.service.impl;

import egovframework.rte.psl.dataaccess.EgovAbstractMapper;   // ★ 표준 DAO 기반
import org.springframework.stereotype.Repository;

@Repository("formDAO")
public class FormDAO extends EgovAbstractMapper {
    public void insertQuestion(QuestionVO vo) {
        insert("formMapper.insertQuestion", vo);   // 매퍼 XML의 SQL ID 호출
    }
    public FormVO selectForm(String formId) {
        return selectOne("formMapper.selectForm", formId);
    }
}
```

### 3.5 MyBatis 매퍼 XML (JPA 엔티티를 SQL로 — 가장 큰 작업)
```xml
<!-- resources/egovframework/mapper/pmsi/form/Form_SQL.xml -->
<mapper namespace="formMapper">
  <insert id="insertQuestion" parameterType="questionVO">
    INSERT INTO form_question (id, section_id, type, title, required, order_index)
    VALUES (#{questionId}, #{sectionId}, #{type}, #{title}, #{required}, #{orderIndex})
  </insert>
  <select id="selectForm" parameterType="string" resultType="formVO">
    SELECT id, owner_id, title, status FROM form WHERE id = #{formId}
  </select>
</mapper>
```
> 학습 포인트: JPA가 자동으로 해주던 SQL을 직접 쓴다. 표준프레임워크/SI 현장은 SQL을
> 명시적으로 통제하는 걸 선호한다(튜닝·감사 용이). cascade·orphanRemoval 같은 JPA
> 편의는 없어지므로, 섹션/질문/옵션 삭제를 DAO에서 명시적으로 처리해야 한다.

### 3.6 Controller (@Resource 주입 + VO)
```java
package egovframework.pmsi.form.web;

@Controller
public class EgovFormController {
    @Resource(name = "formService")
    private FormService formService;

    @RequestMapping(value = "/pmsi/form/addQuestion.do", method = RequestMethod.POST)
    public String addQuestion(QuestionVO vo, ModelMap model) throws Exception {
        formService.addQuestion(vo);
        return "egovframework/pmsi/form/formResult";   // 뷰 이름(JSP 관례)
    }
}
```
> REST로 가려면 `@RestController` + JSON도 가능하나, 표준 관례는 `.do` URL + JSP가 전통적.
> 학습 목적이면 전통 방식을 한 번 경험해보는 것을 권한다.

## 4. 전환 로드맵 (학습 단계별)

표준프레임워크 사상을 단계적으로 익히는 순서. 한 번에 다 바꾸지 말고 모듈 하나씩.

| 단계 | 작업 | 학습 목표 |
|---|---|---|
| 1 | eGovFrame 4.x 개발환경 설치 + Boot 예제 생성 | 표준 프로젝트 구조 체득 |
| 2 | 폼 빌더 1개 모듈 전환(위 예제대로) | 계층형 + ServiceImpl 상속 + MyBatis |
| 3 | 응답 수집 전환 | 순수 로직(QualityJudge) 재사용 경험 |
| 4 | 크레딧 전환 | 트랜잭션·동시성을 표준 방식으로 |
| 5 | 공통 컴포넌트 적용(채번·페이징·예외) | 표준의 진짜 가치 체험 |
| 6 | (선택) 공통 컴포넌트 회원관리/게시판 붙이기 | 표준 생태계 이해 |

## 5. 무엇을 잃고 무엇을 배우나 (정직한 평가)

### 전환하며 잃는 것
- JPA 자동 SQL·cascade → MyBatis 수작업(보일러플레이트 증가).
- 생성자 주입의 불변성·테스트 용이성 → `@Resource` 필드 주입(테스트 약간 불편).
- 헥사고날의 모듈 독립성 → 계층형의 단순함(트레이드오프).
- 최신 Java 21 문법 자유도 → 표준 적용 버전 제약(Spring 5/6 기반).

### 전환하며 배우는 것
- **한국 공공/SI 시장의 표준 사고방식.** 입찰·인증의 실제 요건.
- **계층형 아키텍처의 명료함.** 작은~중간 프로젝트에 충분히 합리적.
- **MyBatis의 SQL 통제력.** ORM이 숨기는 것을 직접 다루는 경험.
- **공통 컴포넌트 재사용.** 채번·페이징·권한을 바닥부터 안 만드는 생산성.

> 핵심 학습 결론: 표준프레임워크는 "더 나은 기술"이 아니라 "표준화된 기술"이다.
> 개인 서비스엔 현재의 현대 Spring Boot가 낫고, 공공 사업엔 표준프레임워크가 필수다.
> 둘 다 다룰 줄 알면 한국 백엔드 시장 어디든 대응 가능하다.

## 6. 권장 학습 순서 정리

1. 이 문서로 **구조 차이**를 머리에 넣는다(§1, §2).
2. eGovFrame 개발환경에서 **Boot 예제 프로젝트**를 뜯어본다.
3. **폼 빌더 하나**를 §3 예제대로 직접 전환한다(가장 많이 배우는 단계).
4. 순수 로직(검증·판정)을 **재사용**하며, 계층만 바뀜을 체감한다.
5. 공통 컴포넌트(채번·페이징)를 적용해 **표준의 생산성**을 경험한다.

> 전환은 "전부"가 아니라 "한 모듈 끝까지"가 학습 효율이 가장 높다.
> 폼 빌더 하나를 완주하면 나머지는 같은 패턴의 반복임을 알게 된다.
