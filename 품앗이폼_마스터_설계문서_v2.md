> 각 설계 결정을 시니어 엔지니어 관점에서 왜 그렇게 했는지 설명한다.
> 최종 갱신 기준: 순수 로직 TDD 누적 **250개 테스트 통과**.

## 0. 이 문서를 읽는 법

품앗이폼은 "남의 설문에 응답해 크레딧을 벌고, 그 크레딧으로 내 설문을 띄우는" 상호 응답
플랫폼이다. 시간이 없는 사람은 크레딧을 현금으로 산다(수익 모델).

이 문서는 세 가지를 분리해서 보여준다:
1. **무엇을 만들었나** — 모듈별 구현 현황과 검증 수준(§3).
2. **왜 그렇게 만들었나** — 핵심 설계 결정과 그 근거(§4~5).
3. **무엇을 안 만들었나** — 미구현과 그 이유(§6).

## 1. 개발 방법론과 환경 (왜 "순수 로직 TDD"인가)

### 1.1 작업 환경의 제약
작업 컨테이너의 egress 프록시가 Maven Central을 차단했다(`host_not_allowed`).
즉 **Spring/JPA를 빌드할 수 없었다.** 이 제약이 전체 개발 방식을 결정했다.

### 1.2 그래서 택한 전략: 프레임워크 분리 TDD
핵심 비즈니스 로직을 Spring에서 분리해 **순수 Java로 작성하고 실제로 RED→GREEN→REFACTOR
사이클을 돌렸다.** Spring 코드는 그 검증된 로직을 감싸는 얇은 어댑터로 작성했다.

이건 제약 때문에 어쩔 수 없이 한 게 아니라, **사실 좋은 아키텍처다.** 비즈니스 규칙이
프레임워크에 묶이지 않으면:
- 테스트가 빠르고 결정론적이다(컨테이너·DB 없이 millisecond 단위 실행).
- 규칙이 Spring/JPA/PostgreSQL 세부에 오염되지 않는다.
- 프레임워크를 바꿔도 핵심 로직은 산다.

대가로, **Spring 통합과 실제 DB 동작은 이 환경에서 검증하지 못했다.** 그래서 모든 문서에
구현 상태를 4단계로 정직하게 표기한다(아래 범례).

### 1.3 구현 상태 범례
| 표기 | 의미 |
|---|---|
| ✅ | 순수 로직 TDD로 실제 검증(RED→GREEN 사이클 통과) |
| 🟡 | Spring 코드 작성 완료, 로직 대조했으나 빌드 미실행 |
| 🟦 | 통합 테스트 코드 작성됨, 실 DB/브로커 실행 대기 |
| ⬜ | 설계만, 코드 없음 |

### 3.1 크레딧 트랜잭션 (화폐 무결성)
크레딧은 사용자 간 거래되는 화폐 → 동시성 버그 = 통화 복제/소실 = 신뢰 붕괴.
- ✅ 자산별 락(예치=비관적 FOR UPDATE, 적립=낙관적 @Version), append-only 원장,
  (reason,ref_id) 유니크 멱등, 정산 원자성. 시뮬레이터로 무결성 붕괴 재현 검증.
- 🟡 Spring(CreditTransactionService) / 🟦 동시성 IT / ⬜ escrow 잠금 실체화

### 3.2 이벤트 측정 인프라 (수신 → 집계 → baseline) — **누적 75 테스트**
실험을 설계하려면 baseline(완료율 등 실측)이 필요하고, 그러려면 이벤트를 안 잃고 모아
집계해야 한다. 측정 인프라를 가장 깊게 판 이유다.
- ✅ 이벤트 수신(검증·멱등·rate limit·세션 경합 복구), 큐 유실방지(재시도·DLQ·멱등),
  배치(사전집계 멱등·타임아웃 마감 경합방어), last_event_at(순서역전·미래값 방어),
  baseline(가중집계·신뢰도) → 표본 크기 산출까지 파이프라인 전체.

## 4. 핵심 설계 결정과 근거 (왜 그렇게 만들었나)

### 4.1 동시성 — "같은 문제, 다른 해법" (가장 중요한 통찰)
이 프로젝트에서 동시성 문제가 **네 번 다른 모습으로** 나타났고, 매번 해법이 달랐다.
표면은 다 "동시성"이지만 경쟁의 성격이 다르기 때문이다.

| 위치 | 경쟁 성격 | 해법 |
|---|---|---|
| 크레딧 예치금 차감 | 같은 행 반복 경쟁(핫) | 비관적 락 (FOR UPDATE) |
| 세션 생성 | 생애 1회 생성 경쟁 | 낙관적 insert-recover-on-conflict |
| 타임아웃 마감 | 마감 직전 상태 변경 | 조건부 UPDATE (WHERE end_state IS NULL) |
| 응답 1인 1회 | 동시 제출 | 유니크 제약 + 충돌 시 DUPLICATE |

**공통 원칙: check-then-act를 원자적 연산으로 만든다.** 하지만 표면이 같다고 해법을
복사하면 안 된다. 핫한 자원은 비관적 락이 맞고, 생애 1회 경쟁은 낙관적이 맞다. 이걸
구분하는 게 이 시스템 설계의 핵심이었다.

### 4.2 멱등성 — 모든 쓰기 경로의 기본기
분산 시스템에서 "정확히 한 번"은 불가능하다. 그래서 **at-least-once + 멱등 소비**를 택했다.
중복이 와도 결과가 같으면 유실보다 낫다.
- 크레딧 정산: (reason, ref_id) 유니크
- 큐 소비: messageId 멱등 + 세션 전이 자체가 멱등
- 사전집계: 전체 재계산 + upsert
- 결제 적립: orderId를 ref_id로 (크레딧 원장과 같은 메커니즘)

같은 멱등 메커니즘이 여러 모듈에서 재사용된다 — 한 번 잘 설계한 패턴의 복리.

### 4.3 측정을 먼저 판 이유 (의존성 역순 개발)
제품 본체(폼 빌더)보다 측정 인프라를 먼저 깊게 팠다. 왜냐하면:
- D4(크레딧 비용 자동 산출), D5(지급률)는 **실험으로 보정**해야 하는 값이다.
- 실험을 설계하려면 baseline(실측 완료율 등)이 필요하다.
- baseline을 내려면 이벤트를 안 잃고 모아 집계해야 한다.
즉 **서비스 경제의 핵심 파라미터가 측정에 의존**하므로, 측정을 먼저 세웠다.

### 4.4 "받느냐"와 "보상하느냐"의 분리 (응답 수집)
응답 유효성(폼 규칙 준수)과 어뷰징(성의 있느냐)은 다른 질문이다. 섞으면 안 된다.
- 유효성 실패 → 제출 거부(데이터도 안 받음).
- 어뷰징 reject → **데이터는 받되 크레딧만 안 줌.** 형식적으로 유효한 데이터를 버리지 않는다.
이 분리 덕에 제작자는 데이터를 다 받고, 크레딧 경제는 무성의를 거른다.

### 4.5 집계는 pass만, 내보내기는 전부 (결과)
같은 응답인데 용도가 달라 필터가 반대다.
- 그래프(집계): 결과 요약 → 불성실 응답 섞이면 왜곡 → pass만.
- 엑셀(내보내기): raw 데이터 제공 → 분석가가 직접 판단 → 전부(quality_flag로 구분).

### 4.6 포트 & 어댑터 (모듈 경계)
모듈 간 의존을 인터페이스(포트)로 역전시켰다. 응답 수집이 크레딧을 직접 모르고
`CreditSettlementPort`만 안다. 결제가 토스를 직접 모르고 `PaymentGateway`만 안다.
효과: 계약 없이 결제 도메인을 Fake로 검증, 모듈을 독립적으로 개발·테스트.

### 4.7 시간 의존은 주입한다 (테스트 결정론)
rate limiter, 작업 만료, last_event_at에서 반복 등장한 교훈. 시스템 시계를 코드에 박으면
테스트가 비결정적이 된다. 시간을 파라미터로 주입받아 결정론적으로 검증했다.
(작업 만료 테스트가 처음 실패한 게 정확히 이 문제였고, 주입으로 해결했다.)

### 4.8 자동은 빠르게, 복구는 신중하게 (가드레일)
가드레일은 위험 감지(중단)는 자동·즉시, 정상 복귀(재개)는 수동·승인으로 비대칭 설계했다.
유의성 게이트로 표본 부족 시 오탐 중단을 막고, 쿨다운으로 중단-재개 반복을 막는다.

## 6. 무엇을 안 만들었나 (미구현과 그 이유)

정직하게 분리한다. 안 만든 데에는 이유가 있다.

### 6.1 환경 제약으로 못 한 것
| 항목 | 이유 |
|---|---|
| Spring/JPA 빌드·실행 | Maven Central 차단 → 순수 로직만 검증 |
| 통합 테스트(Testcontainers) 실행 | 실 DB·Docker 불가 → 코드만 작성 |
| 네이티브 SQL 검증(percentile_cont, ON CONFLICT 등) | 실 PostgreSQL 필요 |
| POI 엑셀 실제 생성 | 의존성 다운로드 불가 → 행렬 변환만 검증, POI 코드는 주석 |

→ 이건 **로컬/CI에서 `./gradlew build` + Testcontainers로 마저 검증**해야 한다.

### 6.2 외부 계약이 필요해 못 한 것
| 항목 | 이유 |
|---|---|
| 실제 PG 연동(결제) | 사업자등록 → PG 계약 → API 키 발급 선행 필요. 도메인·Fake만 완성 |
| 결제 웹훅·정산·세금계산서 | PG 계약 후에만 |

→ 의도적으로 **PG에 의존하지 않게 추상화**해서, 계약 후 어댑터 한 곳만 채우면 되게 했다.

### 6.3 우선순위로 미룬 것 (설계만)
| 항목 | 상태 |
|---|---|
| 인증/인가(소셜 로그인, 1인 1계정) | ⬜ — 제품 핵심 로직 먼저 |
| 분산 rate limit (Redis 전환) | ⬜ — 현재 인메모리 토큰버킷 |
| 관측성(로깅·메트릭·트레이싱) | ⬜ |
| CORS, payload 크기 제한 | ⬜ |
| 각 모듈 포트 구현체(실제 DB 조인) | ⬜ — 인터페이스는 정의됨 |
| escrow 잠금 실체화 | ⬜ — 설계는 D7에 확정 |

### 6.4 사용자가 명시적으로 SKIP한 것
- baseline 확보용 실제 응답자 모으기("현재 단계 비현실적"으로 SKIP).
