# 품앗이폼 — 측정 인프라 상세 설계 v1.0

baseline 확보를 위한 응답자 여정 이벤트 로깅. 출시 시점에 작동해야 하며, 나중에 추가하면
그 기간의 데이터가 통째로 사라진다.

---

## 0. 왜 필요한가

실험의 표본 크기는 baseline(완료율 등)에서 나온다. baseline은 코드가 아니라 실제 행동
데이터에서만 나온다. 그 데이터를 모으려면 응답자의 여정이 이벤트로 기록되고 있어야 한다.

가장 흔한 실패: **완료율을 재려는데 분모가 없다.** 제출은 RESPONSE에 남지만, 시작했다
도중에 떠난 사람은 흔적이 없다. 그러면 완료율(제출/시작)의 "시작"을 셀 수 없다.
→ 응답자 여정 전체를 별도 이벤트 계층으로 남긴다.

---

## 1. 측정할 지표와 정의

| baseline 지표 | 정의 | 쓰임 |
|---|---|---|
| 완료율 (pass 기준) | survey_submitted(quality_flag=pass) / survey_started | 실험 A 표본 계산의 핵심 baseline |
| 제출률 | survey_submitted / survey_started | 참고(불성실 포함) |
| 질문별 이탈률 | 이탈 세션의 마지막 응답 문항 분포 | 폼 빌더 개선 + 이탈 원인 |
| 평균/중앙 소요시간 | submitted_at − started_at | estimated_min 보정의 출발점 |
| 불성실 응답률 | quality_flag in (hold,reject) / submitted | 가드레일 임계의 기준선 |
| 구매 전환율 | 구매 사용자 / 활성 사용자 | 실험 B baseline (크레딧 원장에서 별도 산출) |

주의: 완료율 baseline은 반드시 **pass 기준**. 불성실 응답을 완료로 치면 지표가 오염된다.
(검증 예: 제출률 60%여도 pass 완료율은 50%일 수 있음 — 1건이 reject로 걸러짐)

## 2. 데이터 모델

### 2-1. response_session — 완료율의 분모
한 응답자가 한 설문을 "시작"한 단위. 비로그인 응답자도 추적해야 하므로 anon_session_id 사용.
- 진행중: end_state = NULL
- 종료: end_state in (submitted, abandoned, timeout)
- 제출 시 response_id 연결, quality_flag·duration_sec 기록
- last_question_order = 이탈 지점 분석용

### 2-2. survey_event — 여정 전체 (append-only)
이벤트 5종: survey_viewed / survey_started / question_answered / survey_submitted / survey_abandoned
- survey_started ★ = 완료율 분모. 첫 상호작용 시 1회만.
- question_answered = 질문별 이탈 지점 추적
- CHECK 제약으로 오타 이벤트 차단

### 2-3. form_metrics_daily — 사전집계
배치가 일별로 채운다. 대시보드/실험은 원본이 아니라 이걸 읽어 부하를 분리.

스키마 전문: `V2__measurement.sql` (Flyway 마이그레이션)

## 3. 이탈 감지 (가장 까다로운 부분)

이탈은 "이벤트가 안 오는 것"이라 감지가 어렵다. 2-방식 조합:

### 3-1. 명시적 감지 (클라이언트)
- `visibilitychange`(hidden) + `pagehide`에서 abandoned 비콘 전송
- 반드시 `navigator.sendBeacon()` 사용. 일반 fetch는 페이지 unload 시 취소됨.
- 시작했고(started) 미제출일 때만 전송. 중복 방지 플래그.
- 모바일은 visibilitychange가 pagehide보다 신뢰도 높음 → 둘 다 건다.

### 3-2. 추정 감지 (서버 배치)
- 비콘이 유실될 수 있으므로(네트워크 끊김 등) 백업 필요.
- 배치가 "마지막 이벤트 후 30분+ 무활동 & 미제출" 세션을 timeout으로 마감.
- 임계 30분은 설정값 — 설문 길이에 따라 조정.

→ 명시적(빠르지만 유실 가능) + 추정(느리지만 누락 없음)으로 빈틈을 메운다.

코드: `survey-tracker.js` (클라이언트), 배치 쿼리는 `verify_abandonment.py`에서 검증.

## 4. 집계 쿼리 (검증 완료)

`verify_metrics.py`에서 샘플 데이터로 전 지표 검증 통과:
- 완료율: 시작10 / 제출6 / pass5 → pass 완료율 50%, 제출률 60% ✓
- 질문별 이탈: 1번·3번 문항에서 각 1건 ✓
- 불성실률: 제출6 중 reject1 → 16.7% ✓
- 소요시간: 중앙값 180초, 평균 204초 ✓
- 타임아웃 이탈: 30분+ 무활동 미제출 2건 정확 감지 ✓

핵심 쿼리 형태(완료율):
```sql
SELECT
    count(*) AS started,
    count(*) FILTER (WHERE quality_flag='pass') AS passed,
    round(100.0 * count(*) FILTER (WHERE quality_flag='pass') / count(*), 1) AS pass_rate_pct
FROM response_session WHERE form_id = ?;
```

## 5. 구현 체크리스트 (출시 전 필수)

- [ ] response_session / survey_event / form_metrics_daily 테이블 생성 (V2 마이그레이션)
- [ ] 클라이언트 트래커 탑재: viewed/started/answered/submitted/abandoned
- [ ] sendBeacon 이탈 감지 (visibilitychange + pagehide)
- [ ] 서버 이벤트 수신 API (/api/events) — 인증 불필요(익명 세션 허용), rate limit
- [ ] quality_flag 판정 로직 가동 (불성실률 측정하려면 출시부터 ON)
- [ ] 일별 사전집계 배치 (form_metrics_daily)
- [ ] 타임아웃 이탈 마감 배치 (30분 무활동)
- [ ] experiment_arm 컬럼은 만들어두되 실험 전엔 NULL (나중에 채움)

> 측정 인프라는 "나중에 추가" 불가. baseline 기간 동안 안 쌓이면 그 기간이 날아간다.

---

## 6. 프라이버시 주의
- anon_session_id는 추적용 임의 키. 개인 식별정보 아님.
- IP/device는 어뷰징 감지에만 쓰고 해시 저장(ip_hash).
- 이벤트 meta JSON에 응답 내용(answer value)을 넣지 않는다 — 이벤트는 "행동"만, 내용은 ANSWER 테이블.
