# 품앗이폼 — 피드/매칭 하이브리드 추천 (TDD) v1.0

D1(하이브리드 정산 엔진)을 실제 추천 로직으로 구체화. 풀 방식 기본 + 1:1 매칭 레이어.

---

## 0. 하이브리드 추천이란

품앗이의 순환: 남의 설문에 응답해 크레딧 벌고 → 내 설문을 띄운다. 그래서 "응답할 만한
설문"을 추천하는 피드가 필요. D1의 하이브리드는 두 레이어:
 - 풀 방식(기본): 모든 활성 설문을 후보로 점수 정렬.
 - 1:1 매칭(추천): 내 설문에 응답해준 사람의 설문을 우선 노출(호혜성 촉진).

두 레이어를 점수 합산으로 결합한다.

---

## 1. 자격 필터 (FeedEligibilityTest, 14 테스트)

점수를 매기기 전에 후보가 될 수 있는 설문을 거른다.

| ID | 규칙 |
|---|---|
| E1 | ★자기 설문 제외★ 내가 만든 설문 안 뜸 |
| E2 | ★이미 응답 제외★ 내가 응답한 설문 안 뜸(1인 1회) |
| E3 | ★마감 제외★ ACTIVE 아니면(DRAFT/CLOSED) 안 뜸 |
| E4 | ★예치금 소진 제외★ 남은 escrow < 1응답 보상이면 안 뜸(응답해도 크레딧 못 받음) |
| E5 | ★목표 도달 제외★ maxResponses만큼 모았으면 안 뜸 |
| E6 | ★마감일 지남 제외★ deadline 지났으면 안 뜸 |
| E7 | 다 통과하면 후보 |

> E4가 크레딧 시스템과 연결: escrow 잠금 모델(D7)에서 등록 시 예치한 크레딧이
> 응답마다 차감된다. 바닥나면 응답자가 보상을 못 받으므로 피드에서 빼는 게 정직하다.

## 2. 하이브리드 점수 (FeedRankingTest, 8 테스트)

```
점수 = 풀 기본(R1) + 1:1 매칭 부스트(R2) + 공정성(R3) + 신선도(R4)
```

| ID | 규칙 | 효과 |
|---|---|---|
| R1 | 풀 기본: 보상 클수록 높음 | 응답 유인 |
| R2 | ★1:1 매칭 부스트★ 나에게 응답해준 사람 설문 가산 | 호혜성(D1 핵심) |
| R3 | ★공정성★ 누적 응답 적을수록 가산 | 신규/소외 설문 노출(빈익빈 방지) |
| R4 | 신선도: 최근 설문 소폭 가산 | 오래된 설문 영구 점유 방지 |
| R5 | 최종 점수 내림차순 정렬 | — |
| R6 | ★다양성★ 같은 제작자 maxPerOwner 상한 | 한 제작자 도배 차단 |
| R7 | 상위 K개만 반환 | 페이지네이션 |

> R6 설계 판단(시니어): maxPerOwner를 엄격한 상한으로 두고 초과분을 버린다.
> 보류 후 재투입하면 결국 그 제작자가 피드를 다 채워 다양성 제약이 무의미해진다.
> 다양성을 위해 피드가 짧아지는 걸 감수.

> R2+R3의 긴장: 매칭 부스트(호혜)와 공정성(신규 노출)은 서로 당긴다. 가중치로 균형.
> 호혜만 강하면 끼리끼리, 공정성만 강하면 보상 무의미. 둘을 합산해 조율.

## 3. Spring 구현

### API
```
GET /api/feed?limit=20    내 피드(하이브리드 추천)
```

### 조율 (FeedService.buildFeed, 5단계)
```
1) loadViewer        — 뷰어 컨텍스트(내 ID, 응답 이력)
2) loadActiveCandidates — DB 1차 필터된 ACTIVE 풀
3) filterEligible    — 자격 정밀 필터(E1~E7)
4) MatchContext      — "나에게 응답해준 사람"(reciprocalOwners) 로드
5) rank              — 하이브리드 점수 + 다양성 + 상위 K
```

### 모듈 경계
- `FeedQueryPort` 인터페이스로 의존 역전 — 피드 모듈이 폼·응답·크레딧 모듈 내부
  엔티티에 직접 의존하지 않게(다른 모듈과 동일한 포트 패턴).
- 구현체가 여러 테이블 조인: 후보(form), 응답 이력(survey_response),
  호혜 관계(누가 내 설문에 응답했나), 예치금(credit).
