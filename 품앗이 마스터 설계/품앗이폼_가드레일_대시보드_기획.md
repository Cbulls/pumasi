# 품앗이폼 — 가드레일 모니터링 대시보드 기획서 v1.0

본 문서는 실험설계서 v1.0의 2-3장(가드레일 지표)과 동시성 설계의 무결성 점검을
운영 가능한 대시보드로 구체화한다.

---

## 0. 목적과 원칙

대시보드의 목적은 "예쁜 지표 표시"가 아니라 **시스템이 망가지기 전에 멈추는 것**이다.

설계 원칙 4가지:
1. **가드레일은 작동한다, 보기만 하지 않는다.** 임계 초과 시 자동 대응(실험군 중단/알림)이
   먼저 발동하고, 대시보드는 그 판정 근거를 사람이 확인하는 창이다.
2. **모든 지표는 통제군 대비 상대값.** 절대값은 외부 요인으로 출렁이므로,
   같은 시점 통제군과의 차이(%p)로 봐야 계수 효과만 분리된다.
3. **위험도 순 배치.** 화면을 위→아래로 훑으면 가장 급한 것이 먼저 보인다.
   (알림 → 실험안전 → 경제건강성 → 어뷰징)
4. **자동 vs 수동 구분.** 명확한 위반은 자동 차단, 오탐 위험이 큰 것(자전거래)은
   검토 큐로 보내 사람이 판단.

## 1. 화면 구성 (위→아래 = 위험도 순)

### 상단: 액션 알림 띠
- 가드레일 임박/초과 시 1줄로 표시 + 바로 행동 버튼
- 정상이면 숨김(노이즈 최소화)

### 섹션 ① 실험 안전 가드레일
임계 초과 시 **해당 실험군 자동 중단**. 가장 위에 둔다.

| 지표 | 정의 | 임계 | 대응 |
|---|---|---|---|
| 불성실 응답률 (vs 통제군) | quality_flag in (hold,reject) 비율 차이 | +5%p | 자동 중단 |
| 7일 재방문율 (vs 통제군) | 가입/활성 후 7일 내 재방문 비율 차이 | −3%p | 자동 중단 |
| 응답 중도 이탈률 | 시작 후 미제출 비율 | +4%p | 경고 → 검토 |

### 섹션 ② 크레딧 경제 건강성
| 지표 | 정의 | 정상 범위 | 위험 신호 |
|---|---|---|---|
| 유통 크레딧 총량 | SUM(모든 사용자 잔액) | 완만한 감소 | 급락(벌 길 막힘)/급증(부정 발행) |
| 일일 소각량 | 당일 BURN 합 | 발행 대비 15~20% | 0 근처(소각 미작동) |
| 현금 구매 전환율 | 구매 발생 사용자 / 활성 사용자 | 안정/상승 | 급락(수익 악화) |
| 예치 부족 설문 수 | escrow 소진으로 일시중지된 설문 | 낮게 유지 | 급증(응답자 부족) |

### 섹션 ③ 어뷰징 탐지
| 지표 | 정의 | 대응 |
|---|---|---|
| 최소시간 미달 차단율 | 예상시간 30% 미만 제출 비율 | 자동 차단(정상 동작 확인용) |
| 자전거래 의심 클러스터 | 동일 IP/device 간 상호 적립 패턴 | **검토 큐**(수동 판단) |
| 신고 누적 계정 | 제작자 신고가 쌓인 응답자 | 임계 시 적립률 페널티/정지 |

### 섹션 ④ 시스템 무결성 (동시성 설계 연계)
| 지표 | 정의 | 대응 |
|---|---|---|
| 잔액-원장 불일치 계정 | balance ≠ SUM(ledger.delta) | 즉시 알림(치명) |
| 낙관적 락 재시도율 | 재시도/전체 정산 트랜잭션 | 급증 시 핫계정 락 전략 재검토 |
| 정산 트랜잭션 실패율 | 롤백/타임아웃 비율 | 급증 시 DB 부하 점검 |

## 2. 지표 집계 정의 (SQL 스케치)

### 불성실 응답률 (군별, 일별)
```sql
SELECT r.experiment_arm,
       date_trunc('day', r.submitted_at) AS d,
       AVG((r.quality_flag IN ('hold','reject'))::int) AS bad_rate
FROM response r
WHERE r.submitted_at >= now() - interval '14 days'
GROUP BY 1, 2;
-- 통제군과의 차이 = arm별 bad_rate − control_arm bad_rate (같은 날 기준)
```

### 유통 크레딧 총량 (일별 스냅샷)
```sql
-- 원장 기반(진실의 원천). 사용자 잔액 캐시가 아니라 ledger로 검산
SELECT date_trunc('day', created_at) AS d,
       SUM(delta) OVER (ORDER BY date_trunc('day', created_at)) AS circulating
FROM credit_ledger
WHERE user_id <> 0   -- 시스템(소각) 계정 제외
GROUP BY 1;
```

### 잔액-원장 불일치 점검 (무결성 배치)
```sql
SELECT b.user_id, b.balance AS cached,
       COALESCE(SUM(l.delta), 0) AS from_ledger
FROM credit_balance b
LEFT JOIN credit_ledger l ON l.user_id = b.user_id
GROUP BY b.user_id, b.balance
HAVING b.balance <> COALESCE(SUM(l.delta), 0);
-- 결과가 한 건이라도 있으면 치명적 — 즉시 알림
```

### 자전거래 의심 클러스터
```sql
-- 동일 device_fp/ip_hash를 공유하는 계정들 사이의 상호 적립
SELECT r1.respondent_id AS a, f.owner_id AS b, count(*) AS mutual
FROM response r1
JOIN form f ON r1.form_id = f.id
JOIN response r2 ON r2.respondent_id = f.owner_id
JOIN form f2 ON r2.form_id = f2.id AND f2.owner_id = r1.respondent_id
WHERE r1.device_fp = r2.device_fp OR r1.ip_hash = r2.ip_hash
GROUP BY 1, 2
HAVING count(*) >= 3;  -- 임계는 운영 데이터로 튜닝
```

## 3. 임계값과 자동 대응 규칙

| 트리거 | 조건 | 자동 대응 | 알림 |
|---|---|---|---|
| 실험군 중단 | 불성실 +5%p 또는 재방문 −3%p, 통계적 유의 | 해당 arm 트래픽 0으로, 기본 계수 복귀 | 즉시 (Slack+담당자) |
| 무결성 위반 | 잔액≠원장 1건 이상 | 해당 계정 거래 동결 | 즉시 (최우선) |
| 통화량 이상 | 총량 7일 변화 ±15% 초과 | 신규 발행 일시 점검 모드 | 즉시 |
| 자전거래 | 클러스터 탐지 | 검토 큐 적재(자동 제재 안 함) | 일배치 |
| 소각 미작동 | 일일 소각 0 근처 | 정산 파이프라인 점검 | 즉시 |

자동 대응에는 반드시 **쿨다운과 수동 오버라이드**를 둔다. 오탐으로 멀쩡한 실험이
반복 중단되지 않도록, 자동 중단 후 재개는 사람 승인을 거친다.

## 4. 통계적 주의 (실험설계서 연계)

- 가드레일 판정도 다중비교 대상이다. 매일 들여다보며 "오늘 유의함"으로 멈추면
  거짓 양성(peeking 문제)이 잦다. **순차검정(sequential testing)** 또는
  사전 정의된 체크포인트에서만 중단 판정을 내린다.
- 단, 무결성 위반(잔액≠원장)과 통화량 폭증은 통계가 아니라 결정론적 사고이므로
  즉시 대응한다. 이 둘은 "실험"이 아니라 "버그/공격" 신호다.
- 6장 간섭(SUTVA) 문제로, 공유 통화 지표(통화량)는 군 간 분리가 안 된다.
  통화량은 군별이 아니라 전체 시스템 단위로만 해석한다.

## 5. 기술 구현 메모 (Java/Spring Boot 스택 기준)

- 집계: 배치(Spring Batch 또는 스케줄드 잡)로 5분/일 단위 사전 집계 테이블 적재.
  대시보드는 사전 집계를 읽어 빠르게 렌더(원천 테이블 직접 조회 금지 — 부하).
- 무결성 배치: 잔액-원장 대조는 야간 전체 + 정산 직후 표본 점검 이중화.
- 알림: 임계 초과 이벤트를 메시지 큐로 발행 → Slack/이메일/PagerDuty 연동.
- 시계열 저장: PostgreSQL 집계 테이블로 충분하나, 지표가 늘면 시계열 DB 고려.
- 프론트: 운영 내부 도구이므로 별도 어드민 앱. 차트는 경량 라이브러리.
