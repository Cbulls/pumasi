### 인터랙션 원칙
1. 모든 질문 카드는 그립 핸들로 드래그 정렬 가능
2. 카드 클릭 → 우측 설정 패널이 해당 질문 옵션으로 전환
3. 디바운스 자동저장(2초). "등록하기" 전까지 draft 상태
4. 섹션(페이지) 단위로 접고 펼치기 가능, 진행률 미리보기

```
FORM

  credit_cost(응답 1건 비용), estimated_sec(자동산출),
  target_filter(json: 연령/성별/직업/지역), max_responses, deadline,
  theme(json), created_at

SECTION
  id, form_id, order_index, title, description

  id, section_id, type, title, description, required, multi_select,
  validation(json), media(json), order_index

OPTION
  id, question_id, label, is_etc, image_url, order_index

BRANCH_RULE
  id, question_id, option_id, target_section_id
  (단일선택형만, target은 현재보다 뒤 섹션만)

RESPONSE
  id, form_id, respondent_id, started_at, submitted_at,
  duration_sec, quality_flag(pass/hold/reject), ip_hash, device_fp

ANSWER
  id, response_id, question_id, value(json)
```

### 설계 결정
- 유형별 차이는 `QUESTION.type` + `validation`(JSON)으로 흡수 → 새 유형 추가 시 스키마 변경 불필요
- 응답값은 `ANSWER.value`(JSON) 단일 컬럼 → 단일선택(문자열)·복수선택(배열)·그리드(객체) 모두 수용
- 분기는 `BRANCH_RULE` 테이블로 분리 → (question, option, target_section) 삼중 관계
- `RESPONSE.duration_sec` / `quality_flag` → 크레딧 지급 판단의 근거 (품앗이 특화)

# PART B. 크레딧 시스템 상세 기획

## B-1. 비용 계산식

```
응답 1건 비용(credit_cost) = max(1, round(
    1                              # 기본 비용
  + estimated_min × 1.0            # 시간 가중 (핵심 변수)
  + essay_ratio × estimated_min × 1.5   # 주관식 부담 가산
  + question_count × 0.05          # 문항 수 소폭 반영
))

응답자 적립(reward) = round(credit_cost × 0.8)   # 80% 지급
시스템 소각(burn)   = credit_cost − reward        # 20% 수수료
```

### 계수 산정 근거
- 시간 계수 1.0: "1분 = 1크레딧"을 기준점으로. 응답자가 체감하는 비용 = 시간.
- 주관식 계수 1.5: 주관식은 동일 시간 대비 인지 부담이 커 이탈률↑ → 추가 보상으로 유인.
- 문항 계수 0.05: 시간에 이미 반영되므로 보조적. 0.05로 약하게.
- 모든 계수는 운영 데이터(실제 완료율·이탈률)로 사후 튜닝하는 설정값.

### estimated_min 산출 (제작자 거짓 입력 방지)
제작자가 입력하지 않고 시스템이 문항 유형별 표준 응답시간을 합산:

| 유형 | 표준 시간 |
|---|---|
| 객관식/드롭다운 | 8초 |
| 체크박스 | 12초 |
| 선형배율/평점 | 6초 |
| 단답형 | 20초 |
| 장문형/주관식 | 40초 |
| 날짜/시간 | 10초 |

→ 실제 응답자 중앙 소요시간이 30건 이상 쌓이면 그 값으로 보정.

## B-2. 80% 지급의 이유 (소각 메커니즘)

- 100% 지급 시: A↔B가 서로 무한 응답해도 총 크레딧 불변 → 아무도 크레딧을 사지 않음 → 수익 0
- 20% 소각 시: 거래마다 통화량이 줄어 디플레이션 → 빈자리를 (a) 신규 가입 보너스 (b) 현금 구매가 채움
- 즉 소각률 = 수익 모델의 펌프. 초기엔 15~20%, 데이터 보고 조정.

## B-3. 크레딧 트랜잭션 (이벤트 소싱)

잔액 컬럼 직접 수정 금지. 모든 변동을 append-only 원장에 기록 후 합산.

```
CREDIT_LEDGER
  id, user_id, delta(+/-), reason(signup_bonus/earn_response/
  spend_escrow/refund/purchase/burn), ref_id(response_id 등),
  balance_after, created_at

balance(user) = SUM(delta WHERE user_id = ?)
```

### 응답 1건 발생 시 (원자적 트랜잭션)
```
BEGIN
  검증 통과(quality_flag = pass) 확인
  INSERT ledger(제작자, -credit_cost, 'spend_escrow', response_id)
  INSERT ledger(응답자, +reward,      'earn_response', response_id)
  INSERT ledger(SYSTEM, +burn,        'burn',          response_id)
COMMIT   # 전부 성공 or 전부 롤백
```

### 예치(escrow) 모델
- 설문 등록 시 `credit_cost × max_responses`를 미리 예치(escrow 잠금)
- 응답 1건 확정마다 예치금에서 차감
- 마감/조기종료 시 잔여 예치금 환불(refund)
- 예치 잔액 부족 → 설문 자동 일시중지(피드 노출 중단)

## B-4. 결과 열람 게이트

```
내 설문 결과 열람 요청
  → 게이트 정책 확인
     (정책안 1: 결과 열람에 크레딧 직접 소모하지 않음.
                "응답 수집 = 예치금 차감"으로 이미 비용 지불 →
                열람은 무료, 단 예치를 위해 크레딧이 필요)
     (정책안 2: 열람 자체에 크레딧 소모 — 권장하지 않음, 이중과금 느낌)
  → 권장: 정책안 1. 게이트는 "설문 등록(예치)" 시점에 작동.
          크레딧 부족 → 등록 불가 → 응답하기/구매하기 유도
```

> 핵심: "남의 설문에 응답 → 크레딧 획득 → 그 크레딧으로 내 설문 예치 → 응답 수집/결과 열람"
> 게이트는 결과 열람이 아니라 **설문 등록(예치)** 시점에 거는 것이 경제적으로 더 깔끔하다. (요구사항의 "응답해야 결과를 본다"는 본질을 동일하게 충족: 크레딧 없으면 애초에 설문을 띄울 수 없으므로.)

## B-5. 하이브리드 매칭 (풀 + 1:1)

- 기본: 풀 방식. 내 설문은 조건 맞는 모든 응답자에게 노출
- 레이어: 내 설문에 응답해준 사람의 설문을 내 피드 상단에 우선 추천 → 상호성
- 추천 정렬 점수 = f(보상 크레딧, 소요시간 짧을수록↑, 타깃 일치, 상호응답 여부)

## B-6. 어뷰징 방지 (크레딧 무결성 핵심)

| 위협 | 방어 |
|---|---|
| 고속 무성의 응답 | 최소 응답시간(예상의 30%) 미만 → reject |
| straight-lining | 객관식 전부 동일 선택 패턴 → hold |
| 주관식 쓰레기값 | 반복문자/초단문 감지 → hold |
| 봇/자동화 | 휴대폰 인증, device fingerprint, CAPTCHA |
| 다중계정 자전거래 | IP·디바이스 클러스터링, 동일 클러스터 간 적립 차단 |
| 크레딧 현금화 | 유저간 양도 금지, 환불 불가, 비정상 적립 모니터링 |

- quality_flag = hold → 제작자 검수 후 수동 승인 시 지급
- quality_flag = reject → 미지급 + 응답자 신뢰도 점수 차감
- 신뢰도 누적 하락 → 적립률 페널티 또는 일시 정지

## B-7. 신규 가입 / 콜드스타트

- 가입 보너스: 10크레딧 (약 2분짜리 설문 1개를 100명 못 미치게 띄울 수 있는 정도 — 맛보기)
- 또는 "첫 설문은 50명까지 무료 예치" 형태의 온보딩 크레딧
- 운영 초기엔 시드 설문(운영진/제휴)으로 응답 피드를 채워 닭-달걀 완화

## B-8. 수익 모델 (크레딧 현금 구매)

- 크레딧 패키지 판매 (예: 100크레딧 3,000원 / 500크레딧 12,000원 등 — 가격은 PG 연동 후 확정)
- 게이트에서 "응답하기"(1순위) / "구매하기"(2순위) 노출
- 환불 정책: 미사용 구매 크레딧만 환불 가능, 적립/보너스 크레딧은 환불 불가
- 향후: 프리미엄 구독(고급 분석, 무제한 응답, 로고 제거)
