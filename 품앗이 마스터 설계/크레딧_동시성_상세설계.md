# 품앗이폼 — 크레딧 트랜잭션 동시성 처리 상세 설계 v1.0

백엔드: Java 21 + Spring Boot 3.x + Spring Data JPA + PostgreSQL

---

## 0. 왜 이 설계가 critical 한가

크레딧은 사용자 간 거래되는 화폐다. 동시성 처리가 틀리면 **크레딧이 복제되거나 사라진다**.
이는 단순 버그가 아니라 서비스 신뢰 자체를 무너뜨리는 사고다.

본 설계는 시뮬레이터로 위협을 재현·검증한 뒤 코드로 옮긴 결과물이다.

---

## 1. 재현된 동시성 위협 (검증 완료)

시뮬레이터 시나리오: 제작자 예치금 100에 200건의 응답이 동시에 각 1크레딧 차감 시도.
정상이라면 100건만 성공, 잔액 0, 음수 없음.

| 방식 | 성공 | 최종잔액 | 원장합 | 음수발생 | 판정 |
|---|---|---|---|---|---|
| 락 없음(NONE) | **200** | **59** | **-200** | no | **깨짐** |
| 비관적 락 | 100 | 0 | -100 | no | OK |
| 낙관적 락 | 100 | 0 | -100 | no | OK |

→ 락이 없으면 **잃어버린 갱신(lost update)** 으로 존재하지 않는 크레딧 100이 차감되고
   응답자 200명에게 보상이 지급된다(크레딧 복제). 락 없는 구현은 절대 금지.

---

## 2. 핵심 설계 결정: 자산 특성별 락 전략 분리

"비관적 vs 낙관적"은 양자택일이 아니다. 자산의 동일 행 경쟁 강도로 나눈다.

### 2-1. 제작자 예치금 차감 = 비관적 락 (PESSIMISTIC_WRITE)
- 인기 설문은 짧은 시간에 같은 행(예치금)에 수백 건의 차감이 몰리는 **핫 계정**
- 낙관적 락이면 version 충돌 → 재시도가 경쟁 강도에 비선형 폭증 → 재시도 한도 초과 실패 증가
- `SELECT ... FOR UPDATE`로 행을 잠가 직렬화하는 것이 정확하고 안정적
- 코드: `CreditBalanceRepository.findByIdForUpdate()` (`@Lock(PESSIMISTIC_WRITE)`)

### 2-2. 응답자 적립 = 낙관적 락 (@Version)
- 한 응답자 잔액에 동시에 여러 적립이 몰릴 일은 드문 **콜드 계정**
- 락 대기 비용 없이 `@Version` CAS로 충분
- 드물게 충돌하면 `@Retryable`로 자동 재시도

### 2-3. 격리 수준
- `READ_COMMITTED` (PostgreSQL 기본). 비관적 락이 행을 보호하므로 그 이상 불필요.
- 더 높이면(REPEATABLE_READ/SERIALIZABLE) 직렬화 실패·성능 저하만 늘어난다.

---

## 3. append-only 원장 (이벤트 소싱) — 공통 기반

락 전략과 무관하게 모든 변동은 `credit_ledger`에 INSERT-only로 기록.

### 3-1. 진실의 원천 분리
- `credit_ledger` = 불변 기록(진실). UPDATE/DELETE 금지.
- `credit_balance` = 빠른 조회용 캐시(물질화 뷰). `balance = SUM(ledger.delta)` 항상 성립.
- 배치로 `verifyIntegrity()` 주기 점검 → 캐시와 원장 불일치 시 알림.

### 3-2. 멱등성 (이중 적립 차단) — 2중 방어
1. **1차(애플리케이션)**: `existsByReasonAndRefId()` 사전 체크
2. **2차(DB)**: `(reason, ref_id)` 유니크 제약
   → 동시에 두 트랜잭션이 1차를 통과해도 DB가 한쪽만 커밋. 경쟁 안전.

이것이 왜 중요한가: 응답 제출이 네트워크 재시도/중복 메시지로 두 번 도착해도,
같은 `response_id`에 대한 정산은 단 한 번만 일어난다.

## 4. 정산 트랜잭션 (응답 1건 확정 시)

```
@Transactional(READ_COMMITTED) + @Retryable(OptimisticLockException, 4회, 지수백오프)
settleResponse(creatorId, respondentId, cost, reward, responseId, coeffVersion):

  0. 멱등 체크: 이미 정산된 responseId면 즉시 return
  1. [비관적 락] 제작자 예치금 -cost   → 원장 기록 (SPEND_ESCROW)
                  잔액 부족 시 InsufficientCreditException → 전체 롤백
  2. [낙관적 락] 응답자 잔액 +reward   → 원장 기록 (EARN_RESPONSE)
  3. [비관적 락] 시스템 계정 +burn     → 원장 기록 (BURN)
  → 셋 다 성공해야 커밋. 하나라도 실패하면 전부 롤백.
```

- `burn = cost - reward` (지급률 0.8이면 cost=5 → reward=4, burn=1)
- 최소비용 1크레딧이면 reward=1, burn=0 (소각 없음) — 도메인 테스트로 검증 완료

---

## 5. 예치(escrow) 잠금 모델 — 추가 권장

위 정산은 "응답 1건"의 동시성을 다룬다. 그 상위에 예치 설계가 있다:
- 설문 등록 시 `cost × max_responses`를 미리 예치(escrow 별도 잠금 잔액으로 이동)
- 응답 확정마다 escrow에서 차감 (위 settleResponse)
- 마감/조기종료 시 잔여 escrow 환불(REFUND)
- escrow 부족 → 설문 자동 일시중지(피드 노출 중단)

이렇게 하면 "결과 열람 게이트"가 아니라 "등록 시 예치"에서 크레딧을 거는
요구사항 명세서 B-4의 권장안과 일치한다.

## 6. 검증 결과

### 6-1. 도메인 규칙 (순수 Java 단위 검증, 5/5 통과)
- 정상 차감 / 음수 잔액 방어 / 소각 계산(r=4,b=1) / 정산 보존(합=0) / 최소비용(b=0)

### 6-2. 동시성 무결성 (Testcontainers 통합 테스트)
- `CreditConcurrencyIT`: 실제 PostgreSQL에 200건 동시 정산 → 100건만 성공, 잔액 0, 음수 없음
- 중복 responseId 정산 → 1건만 반영 (멱등성)
- (CI 환경에서 Docker 필요. 로컬/CI에서 실행)

## 7. 운영 시 주의사항

- **데드락 방지**: 여러 행을 잠글 때 항상 동일한 순서로 잠근다(예: user_id 오름차순).
  현재 설계는 creator → respondent → system 순서 고정. 교차 트랜잭션 데드락 주의.
- **재시도 한도**: 낙관적 락 재시도는 반드시 상한(4회). 무한 재시도 금지.
- **락 타임아웃**: 비관적 락에 `jakarta.persistence.lock.timeout` 설정으로 무한 대기 방지.
- **계수 버전 기록**: 원장에 `coeff_version` 저장 → 실험 중 계수 변경 추적(실험설계서 7장).
- **시스템 계정**: 소각분이 모이는 계정. 통화량 모니터링·이상 적립 감사에 사용.

## 8. 트레이드오프 요약

| 항목 | 비관적 락 | 낙관적 락 |
|---|---|---|
| 적용 대상 | 제작자 예치금(핫) | 응답자 적립(콜드) |
| 경쟁 심할 때 | 대기하지만 안정적 | 재시도 폭증, 실패 위험 |
| 경쟁 약할 때 | 불필요한 락 비용 | 가장 빠름 |
| 데드락 | 가능(순서 고정 필요) | 없음 |
| 구현 복잡도 | 낮음 | 재시도 로직 필요 |

> 결론: 단일 방식 강요 대신 자산별 분리. 원장(이벤트 소싱)은 양쪽 공통 기반.
