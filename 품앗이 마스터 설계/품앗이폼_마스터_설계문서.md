## 1. 한 장 요약

품앗이폼은 구글폼·네이버폼 수준의 설문 제작 기능에 **상호 응답(품앗이) 구조**를 결합한 플랫폼이다.
기존 폼 서비스의 가장 큰 고통은 "설문은 쉽게 만들지만 응답자를 못 모은다"는 것. 품앗이폼은
**응답 크레딧**이라는 화폐로 이 문제를 구조적으로 푼다. 남의 설문에 응답해 크레딧을 벌고,
그 크레딧으로 내 설문을 띄운다. 시간이 없는 사용자는 크레딧을 현금으로 살 수 있고, 이것이 수익 모델이다.

핵심 설계 긴장은 세 가지다:
- **경제 균형**: 크레딧을 너무 후하게 주면 아무도 안 사고(수익 0), 너무 박하게 주면 응답 동기가 꺾인다 → 소각률로 조율.
- **응답 품질**: 품앗이는 본질적으로 성의 없는 응답을 유발한다 → 어뷰징 방지가 MVP 필수.
- **시스템 무결성**: 크레딧은 화폐이므로 동시성 버그 = 통화 복제/소실 = 신뢰 붕괴 → 락 전략 + 원장.

- **D7 (권장): 게이트를 "결과 열람"이 아니라 "설문 등록(예치)" 시점에.**

## 4. 제품 개요와 핵심 차별점

### 4.1 경쟁 서비스 분석 (벤치마킹 결과)
- **구글폼**: 12종 질문 유형, 단일선택형 한정 섹션 분기, 퀴즈 채점, AI 질문 추천.
- **네이버폼**: 객관식·서술식·평점 등, 질문별 필수/복수/분기, 최대 200문항, 분기는 단일선택형·뒤 페이지만.

→ 품앗이폼은 이들의 핵심 제작 기능을 따라가되, **품앗이 + 현금 구매**로 차별화한다.

### 4.2 핵심 메커니즘
```
남의 설문 응답 → 크레딧 적립 → 내 설문 예치/등록 → 응답 수집 → 결과 열람
                    ↑                                      ↓
              (가입 보너스 / 현금 구매)              (20% 소각으로 통화량 감소)
```

## 6. 크레딧 시스템 설계

### 6.1 비용 계산식
```
응답 1건 비용 = max(1, round(
    1                                    # 기본
  + estimated_min × 1.0                  # 시간 가중 (핵심 변수, 앵커=1)
  + essay_ratio × estimated_min × 1.5    # 주관식 부담 가산
  + question_count × 0.05                # 문항 수 (이중계상 방지로 약하게)
))
응답자 적립 = round(비용 × 0.8)            # 80% 지급
소각        = 비용 − 적립                  # 20% 수수료
```

계수의 근거(요약): 시간 계수 1.0은 "1분=1크레딧" 단위 앵커라 옳고 그름이 없다.
주관식 1.5는 인지부담 이론에서 방향만, 크기는 추측 — 실증 보정 1순위.
문항 0.05는 시간에 이미 반영되므로 의도적으로 약하게(이중계상 방지).
지급률 0.8(소각 20%)은 통화 안정성에서 역산한 제약조건. → 전부 운영 config로 분리.

### 6.2 estimated_min 자동 산출 (어뷰징 방지)
제작자 입력 금지. 문항 유형별 표준시간(객관식 8초, 주관식 40초 등) 합산 →
실제 응답 30건↑ 쌓이면 중앙값으로 보정.

### 6.3 80% 지급(소각)의 의미
100% 지급 시 유저끼리 무한 순환만 일어나 아무도 안 산다(수익 0). 20% 소각이 통화량을
서서히 줄여 빈자리를 가입 보너스+현금 구매가 채운다. 소각률 = 수익 모델의 펌프.

### 6.4 하이브리드 매칭 (D1)
기본 풀 방식 + "내 설문에 응답해준 사람의 설문을 우선 추천"하는 매칭 레이어.

### 6.5 게이트 위치 (D7 권장)
게이트를 결과 열람이 아니라 **설문 등록(예치)** 시점에 건다. 크레딧 없으면 설문을 못 띄우므로
"응답해야 내 설문이 돌아간다"는 본질은 동일하게 충족되며 이중과금 느낌이 없다.

### 6.6 어뷰징 방지 (MVP 필수)
최소 응답시간 검증, straight-lining/쓰레기값 탐지, 주의 검증 문항, 신고·평판,
휴대폰 인증·device fingerprint·CAPTCHA, 자전거래 클러스터링, 크레딧 양도/현금화 금지.

## 7. 크레딧 동시성 처리 (구현)

크레딧은 화폐다. 동시성 버그 = 통화 복제/소실 = 신뢰 붕괴. 시뮬레이터로 위협을 재현 검증했다.

### 7.1 재현된 위협
예치 100에 200건 동시 차감 시뮬레이션: **락 없으면 200건 전부 성공 + 원장 -200**
(잃어버린 갱신으로 크레딧 복제). 비관적/낙관적 락은 둘 다 정확히 100건만 성공.

### 7.2 핵심 결정: 자산별 락 전략 분리 (D6)
- **제작자 예치금 차감 = 비관적 락**(`SELECT FOR UPDATE`). 인기 설문은 같은 행에 차감이 몰리는
  핫 계정 → 낙관적 락이면 재시도 폭증. 행 잠금으로 직렬화.
- **응답자 적립 = 낙관적 락**(`@Version`). 한 명에 동시 적립은 드문 콜드 계정 → 락 대기 불필요.
- **공통: append-only 원장**. balance는 캐시, 진실은 ledger. `(reason, ref_id)` 유니크 제약으로
  멱등성 보장(같은 응답 두 번 도착해도 1회만 정산).

### 7.3 정산 트랜잭션
한 `@Transactional` 안에서 제작자 -cost / 응답자 +reward / 시스템 +burn 이 전부 성공하거나
전부 롤백. 낙관적 락 충돌 시 `@Retryable`(최대 4회, 지수 백오프). 데드락 방지 위해 락 순서 고정.

### 7.4 검증 상태 (정직한 한계)
- 도메인 규칙(음수 방어·소각·보존): 순수 Java 단위 테스트 5/5 통과 ✅
- 동시성 무결성: 순수 Java 시뮬레이터로 재현 검증 ✅
- Spring/JPA 코드 컴파일 + Testcontainers 통합테스트: **이 환경에서 미실행** ⚠️
  (egress 프록시 allowlist가 Maven Central을 차단 — 8.x 참조). 로컬/CI에서 실행 필요.

상세: `크레딧_동시성_상세설계.md` / 코드: `credit-concurrency/`

### 7.5 환경 제약 메모
샌드박스 egress 프록시가 도메인 allowlist 방식(`x-deny-reason: host_not_allowed`)이라
Maven Central·Gradle 저장소·Docker 레지스트리가 막혀 Java 빌드 불가. PyPI/npm/GitHub만 허용.
→ 코드 자체는 표준 Spring Boot 관용구이므로 사용자 로컬/CI에서 정상 빌드·테스트된다.

## 9. 데이터 모델

```
FORM(id, owner_id, title, status, credit_cost, estimated_sec, target_filter,
     max_responses, deadline, theme)
 └─ SECTION(id, form_id, order_index, title)
     └─ QUESTION(id, section_id, type, title, required, multi_select, validation, order_index)
         ├─ OPTION(id, question_id, label, is_etc, order_index)
         └─ BRANCH_RULE(id, question_id, option_id, target_section_id)   # 단일선택·뒤섹션
FORM ─ RESPONSE(id, form_id, respondent_id, duration_sec, quality_flag, ip_hash, device_fp)
        └─ ANSWER(id, response_id, question_id, value:json)

CREDIT_BALANCE(user_id, balance, version)                    # 캐시 (@Version 낙관적 락)
CREDIT_LEDGER(id, user_id, delta, reason, ref_id, balance_after, created_at, coeff_version)
              # append-only, UNIQUE(reason, ref_id) = 멱등성
```

설계 결정: 질문 유형 차이와 응답값을 JSON 컬럼으로 흡수(확장성). 분기는 별도 테이블.
`RESPONSE.duration_sec/quality_flag`는 크레딧 지급 판단 근거(품앗이 특화).

## 10. 실험: 크레딧 계수 보정

출시 후 추정 계수를 실데이터로 보정하는 A/B 설계.

- **1순위**: 주관식 가중 W_essay (1.0 / 1.5 / 2.0) — 근거가 가장 약함. 지표=완료율.
- **2순위**: 지급률 (0.70 / 0.80 / 0.85) — 응답 동기(H2) vs 구매 전환(H3)이 상충. 지표=주간 응답 수.
- 시간 계수 1.0은 앵커라 실험 제외.
- **표본 크기**: Java 계산기로 산출. 2군·완료율60%·MDE5%p → 변형군당 ~1,468건,
  3군(본페로니 보정)이면 ~1,778건. baseline은 파일럿으로 교체 필수.
- **최대 난점**: 공유 통화 간섭(SUTVA 위반). 처치군 활동이 통제군 경험을 오염 →
  switchback 또는 클러스터 무작위화 검토. 단순 사용자 단위 배정은 유효 표본을 줄인다.
- 통계 위생: 다중비교 보정, burn-in 제외, 사전등록, peeking 방지.

상세: `품앗이폼_크레딧계수_실험설계.md` / 계산기: `SampleSizeCalculator.java`

## 11. 가드레일 모니터링

목적은 "지표 표시"가 아니라 **망가지기 전에 자동으로 멈추는 것**.

### 11.1 4계층 (위험도 순)
1. **실험 안전**: 불성실 응답률(+5%p)·재방문율(−3%p) 초과 시 해당 실험군 자동 중단.
2. **경제 건강성**: 유통 크레딧 총량(급락/급증), 일일 소각량, 구매 전환율, 예치부족 설문수.
3. **어뷰징**: 최소시간 미달 차단율, 자전거래 의심(검토 큐=수동), 신고 누적.
4. **시스템 무결성**: 잔액≠원장(치명·즉시), 낙관적 락 재시도율, 정산 실패율.

### 11.2 핵심 원칙
- 모든 지표는 **통제군 대비 상대값**(외부 요인 출렁임 분리).
- **통계 판정 vs 결정론 사고 구분**: 실험 가드레일은 순차검정으로 peeking 방지.
  무결성 위반·통화량 폭증은 통계가 아니라 버그/공격 신호 → 1건이라도 즉시 대응.
- 자동 대응엔 쿨다운+수동 오버라이드. 오탐 반복 중단 방지.

상세: `품앗이폼_가드레일_대시보드_기획.md` (집계 SQL 포함)
